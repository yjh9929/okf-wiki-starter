LLM wiki

LLM Wiki: 모을 때 미리 정리해 두기

Andrej Karpathy가 2026년 4월에 공개한 메모에서 나온 방식입니다. 자료가 들어오는 시점에 한 번 정리해서 위키로 만들어 둡니다.

두 방식 비교

RAG와 LLM Wiki, 무엇이 다른가

RAG · 물어볼 때 찾기

  • 질문 100번 = 정리 100번
  • 어제의 답이 오늘 남아 있지 않음
  • 쓸수록 비용만 늘어남
vs

LLM Wiki · 모을 때 정리

  • 정리 1번 + 질문 100번
  • 좋은 답은 위키에 다시 쌓임
  • 쓸수록 위키가 두꺼워짐
Karpathy는 이 위키를 영속적이고 복리로 누적되는 산출물이라고 표현했습니다.
저장 구조

정보를 세 층으로 나눠 담습니다

각 층이 맡는 역할을 확인합니다. 둘은 폴더, 하나는 규칙 문서입니다.

1층

raw/ · 원본

모은 자료를 그대로 둡니다. 수정하지 않습니다. 출처를 되짚는 기준이 됩니다.

2층

wiki/ · 정리본

AI가 만들고 관리합니다. 개념 하나가 파일 하나가 되고, 링크로 이어집니다.

3층

규칙 · 문서

어떤 형식으로 정리할지 적어 둔 문서입니다. 오늘은 okf-wiki 스킬이 맡습니다.

wiki/index.md는 전체 목차, wiki/log.md는 작업 기록입니다.
오늘 할 일

오늘은 세 가지 작업을 합니다

정리

Ingest

raw/의 자료를 읽어 개념 문서를 만듭니다. 같은 개념은 하나로 합칩니다.

질문

Query

위키를 근거로 답하고 출처를 답니다. 쓸 만한 답은 다시 저장합니다.

점검

Lint

끊긴 링크, 고립된 노트, 빠진 표식과 출처를 찾아냅니다.

셋 다 한국어 문장으로 부탁하면 됩니다. 외워야 할 명령어는 없습니다.

OKF 규격

OKF는 마크다운입니다

Google이 2026년 6월에 공개한 파일 형식입니다. 평소 쓰던 마크다운 맨 앞에 표식 몇 줄을 얹은 것이 전부입니다.

규격이 짧은 이유

필수 항목은 type 하나뿐

하나만 필수인 이유

규칙이 많으면 아무도 지키지 않습니다. 최소한만 강제하고 나머지는 열어 둡니다.

나머지 항목

title, description, resource, tags, timestamp. 자기만의 항목을 추가해도 됩니다.

tool도구나 프로그램을 다루는 문서
query-result질문에 대한 답을 저장한 문서
