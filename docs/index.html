Part 1 · 문제 정의

먼저, 요즘 방식 — RAG

RAG(검색 증강 생성)는 "AI에게 내 문서를 붙여 답하게" 하는 가장 흔한 방식입니다.

문서를 통째로 넣을 수 없으니(컨텍스트 한계), 필요한 조각만 찾아서 그때그때 넣어줍니다. 이 "찾아 넣기"가 어떻게 동작하는지 봐야 한계가 보입니다.

RAG 동작 원리

조각내고 → 벡터로 → 질문 때 유사한 걸 검색

문서원본
→
청킹조각으로 쪼갬
→
임베딩조각→벡터(숫자)
→
벡터 DB저장
질문할 때마다 ↓
질문→벡터같은 방식
→
유사도 top-k가까운 조각 검색
→
컨텍스트 주입프롬프트에 붙임
→
LLM 생성

핵심: 정리는 안 하고 저장만 해둡니다. 질문마다 "가까운 조각"을 새로 꺼내 붙일 뿐이죠.

RAG의 한계 ① 비용

질문할 때마다, 매번 다시 읽는다

같은 문서 조각을 질문 100번이면 100번 다시 꺼내 컨텍스트에 붙입니다. 매번 그만큼의 토큰이 다시 청구되죠.

비용 ≈ (질문 수) × (매번 붙이는 조각 크기) — 질문이 쌓일수록 선형으로 증가

"정리해 두면 한 번이면 될 일"을 매번 반복하는 구조입니다.

RAG의 한계 ② 정합성

조각끼리는 서로를 모른다

검색된 조각들은 파편일 뿐, 누구도 "이 조각과 저 조각이 모순인지" 맞춰보지 않습니다.

예) 한 문서엔 "A는 B다", 다른 문서엔 "A는 B가 아니다" → RAG는 둘 다 던져주고, 그 위에서 그럴듯한 환각이 생깁니다. 지식이 통합·검증되는 시점이 없기 때문이죠.

발상의 전환

읽을 때 조립하지 말고,
모을 때 컴파일하자

질의 시점(Query Time)이 아니라 수집 시점(Compile Time)에 미리 정제·통합·연결해 둡니다. — Andrej Karpathy, "LLM Wiki"

컴파일러 비유

지식을 "컴파일"한다는 것

프로그래밍지식 위키하는 일
소스 코드raw/ 원본사람이 모은 불변 입력
컴파일러LLM (에이전트)읽고 정제·통합해 출력
실행 파일wiki/ OKF 문서정제·연결된 산출물
테스트lint무결성 검사
런타임query질의응답

한 번 잘 컴파일해 두면, 이후엔 "실행 파일"만 쓰면 됩니다.

왜 이득인가

비용이 "누적"에서 "1회+α"로

구분RAG (쿼리 타임)LLM Wiki (컴파일 타임)
정제 시점없음 (저장만)수집 때 1회 컴파일
질문 1회 비용원문 조각 재검색·주입작은 인덱스·개념만 로드
N번 질문≈ N × 재검색 (선형 누적)≈ 1 컴파일 + N × 소량
모순·환각통합 없음 → 취약쓸 때 통합·검증 → 견고

질문이 많아질수록(=실제 지식 활용) 격차가 벌어집니다. "미리 한 번"이 이깁니다.

질문할 때마다 조립하지 말고,
모을 때 미리 컴파일하라.

Part 2 · 구조 원리

컴파일 결과물의 정체 — OKF 파일 한 장

--- 여기부터 YAML frontmatter (기계가 읽는 '계약') --- type: core-concept # ★ 필수: 이 문서가 '무엇'인지 title: "OKF" resource: "raw/okf.md" # 출처(어느 원본에서 왔나) tags: [standard] --- # OKF 마크다운 + 최소 YAML 표준. (raw/okf.md) ← 사실마다 출처 관련 개념: [[wiki/patterns/llm-wiki]] ← 그래프 '엣지'

그냥 마크다운입니다. 위쪽 몇 줄(YAML)이 사람+기계 모두 읽는 메타데이터 계약일 뿐.

최소주의 설계

왜 type 하나만 필수일까

type은 "이 문서가 개념인지·인물인지·표준인지"를 기계가 분류·질의할 수 있게 하는 최소 계약입니다.

나머지는 자유 — 생산자는 필요한 키를 마음대로 추가하고, 소비자는 모르는 키를 거부하지 않고 무시합니다. 이 "느슨한 계약" 덕분에 도구가 달라도 서로 읽히는 상호운용성이 생깁니다.

지식 = 그래프

파일은 노드, 링크는 엣지

⬤
노드 = 개념 파일
개념 하나 = 파일 하나. 경로가 곧 고유 ID.
↗
엣지 = [[wikilink]]
개념 간 연결. Obsidian 그래프의 선.
◎
루트 = index.md
모든 노드는 여기서 도달 가능해야 함.

이렇게 보면 뒤에 나오는 "점검(Lint)"이 사실 그래프 무결성 검사라는 게 자연스럽습니다.

아키텍처

입력 · 출력 · 규칙, 딱 3층

📥
raw/ — 입력
사람이 모은 원본. 불변(컴파일러가 소스코드를 안 고치듯).
📚
wiki/ — 출력
LLM이 만든 OKF 문서. 재생성·수정 가능.
📐
schema — 규칙
어떻게 컴파일할지 정한 규칙 문서.

재현성의 열쇠

schema = 컴파일러 설정

LLM은 그냥 두면 매번 다르게 씁니다. schema(Claude Code의 CLAUDE.md 또는 okf-wiki 스킬)가 출력 규칙을 제약합니다.

"모든 문서에 type을 넣어라 · 고아를 만들지 마라 · 사실엔 출처를 달아라" — 이 규칙이 있어서 결과가 일관되고 재현 가능해집니다. gist가 schema 예시로 콕 집어 CLAUDE.md를 든 이유죠.

Part 3 · 내부 동작

세 동작의 속을 열어보자

Ingest 정리 · Query 질문 · Lint 점검 — 각각 안에서 무슨 일이 일어나는가

Ingest · 내부 동작

원본을 개념으로 "컴파일"

raw 읽기
→
개념 추출무엇이 핵심인가
→
중복 병합같은 개념 합침
→
개념=파일원자화
→
[[상호링크]]
→
index·log 갱신

여러 원본에 흩어진 같은 개념을 하나로 합치며 출처를 병기합니다. 이 "합치는 순간"이 RAG엔 없던 단계예요.

핵심 통찰

모순은 "쓸 때" 잡힌다

병합하려면 LLM이 두 문장을 같은 자리에서 비교해야 합니다. "A는 B다"와 "A는 B가 아니다"가 만나면 그 순간 충돌이 드러나죠.

RAG는 읽을 때(read-time) 조각을 던질 뿐 비교하지 않습니다. LLM Wiki는 쓸 때(write-time) 통합·검증합니다 — 그래서 결과가 더 일관됩니다.

Query · 내부 동작

원문이 아니라 "정제된 위키"에 묻는다

질문
→
인덱스+개념페이지 로드원문 전체 X
→
합성 + 출처 인용
→
역보관좋은 답을 다시 저장

RAG처럼 매번 원문 조각을 재검색하지 않습니다. 이미 정리된 요약만 보니 싸고, 출처가 붙어 있어 검증됩니다.

Lint · 내부 동작

점검 = 그래프 무결성 검사

① type 누락
모든 wiki 문서에 필수 계약(type)이 있나 — 스키마 준수.
② 깨진 링크
[[link]]가 실제 파일을 가리키나 — 엣지 유효성.
③ 고아 문서
index에서 도달 불가한 노드 — 도달성(BFS).
④ 출처 누락
사실에 (raw/..) 근거가 붙었나 — 추적성.

제공 lint_okf.py가 이 4가지를 자동 판정합니다(PASS/FAIL). 컴파일러의 "테스트"에 해당.

Compounding

쓸수록 똑똑해지는 이유

질문·분석
→
OKF로 역보관
→
노드·엣지 ↑
→
그래프 밀도 ↑
→
다음 질문이 더 정확

지식이 단조 증가합니다. RAG는 아무리 질문해도 저장소가 그대로지만, 위키는 매 질문이 자산이 됩니다 — 이게 "복리".

Part 4 · 도구

그럼 "컴파일러"는 누가? — Claude Code

Claude Code는 파일을 읽고 쓸 수 있는 에이전트입니다. read → reason → write 루프를 돌며 raw를 읽고 wiki를 씁니다.

API를 코드로 부르는 게 아니라, schema(CLAUDE.md)를 지키며 대화로 지시하면 됩니다. 본인 구독으로 동작해 대화형 작업엔 API 종량 과금이 없습니다.

okf-wiki 스킬

규칙을 "스킬"로 포장한 이유

앞의 schema 규칙(type 필수·고아 금지·출처)을 okf-wiki 스킬 하나에 담았습니다.

학생은 규칙을 외울 필요 없이 설치만 하면, 매번 같은 품질로 컴파일됩니다 — 재현성 확보. 외부 유료 도구·API 키도 필요 없습니다.

환경 세팅

6단계면 준비 끝

  1. Claude Code 설치·로그인
  2. 실습 repo 내려받기 (말로)
  3. Obsidian 설치·열기
  4. 웹 클리퍼 확장 [필수]
  5. 플러그인 3개 [추천]
  6. 확인 → 첫 실습

상세는 세팅_가이드.md. Node.js는 필요 없어요(심화 때만).

한 장 요약

원리 3줄

  • RAG는 읽을 때 조립 → 반복 비용·모순
